Хардкодити паролі чи інші чутливі дані - погана практика. Винесення чутливих даних у файл .env є одним із варіантів більш коректного підходу, тож я реалізував його у себе, та стикнувся з декількома проблемами при передачі змінних в PostgreSQL (це має більше підводних каменів, ніж передача змінних у звичайні bash-скрипти).

**Проблема 1: Неправильний експорт змінних**

Процес перенесення змінних у PostgreSQL вимагає кількох кроків:
1. Перенесення змінних у середовище, у якому відбувається робота з psql.<br>
Якщо спробувати завантажити змінні з .env файлу, обгорнувши команду в лапки, Bash сприйме весь вивід як один довгий рядок, а не як набір окремих змінних.<br>
**Неправильно:<br>**
`export "$(grep -v '^#' database.env | xargs)"`<br>
В результаті psql отримує невалідний запит, що призводить до синтаксичної помилки.<br>
**Правильний спосіб передбачає використання команди без лапок:<br>**
`export $(grep -v '^#' database.env | xargs)`

2. Перенесення змінних з середовища у psql, яке відбувається приблизно наступним чином:<br>
`sudo -u postgres psql \`<br>
`  -v user_name="$USER" \`<br>
`  -v user_password="'$PASSWORD'" \`<br>
`  -f "$WORKING_DIR/init.sql"`<br>
3. Використання змінних в sql-файлі:<br>
`CREATE USER :user_name WITH PASSWORD :user_password;`<br>


**Проблема 2: Прихований символ переносу каретки (\r)**

Я зіткнувся з проблемою, що при абсолютно ідентичному заданні зовнішніх змінних, фронтенд-сервіс отримував помилку, що вказувала на некоректність пароля до бази даних. <br>
Це пов'язано із різним способом закінчення рядків в ОС Windows та Linux. Дана проблема поширена загалом при розробці на Windows, та розгортанням у Linux. <br>
Якщо файл .env створювався або редагувався у Windows, він використовує закінчення рядків CRLF (\r\n). Linux працює з форматом LF (\n). При експорті такого файлу в Linux-середовище, невидимий символ \r може залишитися як частина змінної. Наприклад, в конкретно моєму випадку PostgreSQL зберіг змінну у початковому вигляді (наприклад, password\r). Коли інший сервіс (у моєму випадку Systemd) намагається підключитися до бази, він відділяє символ переносу каретки (\r) від значення змінної. Через цю розбіжність виникла помилка автентифікації.

**Рішення:**

Очищення отриманих змінних від зайвих символів:
`tr -d '\r'`

Додатково можна використати файл .gitattributes, який вказує Git, як зберігати файли. Приклад для зберігання файлів з правильними переносами:<br>
`*.env text eol=lf`<br>
`*.sh text eol=lf`<br>
`*.sql text eol=lf`
