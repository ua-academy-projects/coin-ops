### Ключові висновки по попередньому завданню

1. **Один Terraform root** — не два, не три. Один `main.tf` який читає конфіг і вирішує що піднімати.

2. **Один конфіг-файл** — не окремий YAML для AWS і окремий для GCP. Один файл, в якому є і загальні параметри, і перелік VM, і вказівка на клауд.

3. **Словник розмірів** — замість прямого зазначення `machine_type: e2-micro` краще зробити:
   ```yaml
   sizes:
     small:
       gcp: "e2-micro"
       aws: "t3.micro"
   vms:
     jump-host:
       size: small
   ```
   Terraform читає словник і підставляє правильний тип для потрібного клауду.

4. **Логіка вибору клауду всередині модуля** — не зовні. Модуль сам перевіряє змінну і вирішує чи йому взагалі щось робити.

5. **Internal IP виводити в outputs** — якщо VM мають між собою комунікувати, їхні IP/hostname треба виводити з модуля і записувати в конфіг або Ansible inventory.

6. **Дефолтні значення в модулі** — якщо параметр не вказаний для конкретної VM, модуль має брати дефолт, а не падати.

7. **Не хардкодити нічого в Terraform** — ні IP, ні machine type, ні імена. Все в конфіг-файлі.

## Частина 2 — Нова задача: задеплоїти аплікацію на хмарну інфраструктуру

### Загальний опис

Підняти реальну аплікацію (ваш проект з академії) на GCP-інфраструктурі яку ви вже створили через Terraform. Аплікація має бути в Docker-контейнерах, розподілена по VM, доступна через домен з HTTPS-сертифікатом.

### Вимоги

- Мінімум 2 VM з контейнерами (щоб була комунікація між різними машинами)
- 1 VM з базою даних (PostgreSQL або те що у вашому проекті)
- Аплікація упакована в Docker (образи вже є, ви вже вмієте)
- Ansible налаштовує VM після того як Terraform їх підняв
- Ansible inventory має знати IP/hostname кожної VM

### Що треба зробити по кроках

**Крок 1 — Налаштувати Ansible inventory**

Після `terraform apply` потрібно знати які машини існують і куди підключатися. Два варіанти:
- Прописати IP з terraform output вручну в статичний inventory (прийнятно як перший етап, але не ідеально)
- Зробити динамічний inventory: Terraform записує outputs, скрипт або плагін Ansible читає ці дані і генерує inventory автоматично

Ментор назвав динамічний inventory "шикарна практика" і рекомендує його зробити.

**Крок 2 — Ansible конфігурує VM**

Після того як inventory готовий, Ansible має:
- Встановити Docker на кожну VM
- Розкласти Docker Compose файли або запустити контейнери

**Крок 3 — Отримати домен**

- Зареєструватися на pp.ua — видають безкоштовні доменні імена в зоні `.pp.ua`
- Зареєструватися на Cloudflare — безкоштовний акаунт
- Перенести домен на Cloudflare nameserver — управляти DNS через Cloudflare

**Крок 4 — Отримати SSL-сертифікат**

Інструмент: **Certbot** (від Let's Encrypt). Безкоштовні сертифікати.

Метод валідації: **DNS challenge через Cloudflare**
- Certbot додає TXT-запис в DNS для підтвердження що ти власник домену
- Перевага: не потрібно відкривати порт 80/443 назовні під час валідації
- Сертифікат генерується локально

Важливо: спочатку тестувати на **staging** (`--staging` флаг в certbot) — Let's Encrypt має rate limit на кількість видачі реальних сертифікатів на добу. Staging видає невалідні сертифікати (браузер покаже попередження), але механізм той самий. Коли впевнений що все працює — перемикатися на production.

**Крок 5 — Виставити аплікацію назовні**

Два варіанти:
- З load balancer: можна слухати на порту 80, балансер приймає запити і перенаправляє
- Без load balancer: certbot + nginx/аплікація безпосередньо на порту 443

Ментор: якщо немає load balancer то тільки порт 443 з сертифікатом. Порт 80 без load balancer — не рекомендується.

### Що НЕ треба робити зараз

- Docker Swarm — ментор сказав "лишня робота", в дорослих проектах не використовується, якщо є потреба в оркестрації одразу беруть Kubernetes
- Autoscaling groups — не обов'язково для цієї задачі
- Packer (збірка власних образів) — цікаво, але не для цього етапу

### Tailscale (бонусна тема від ментора)

Ментор окремо розповів про Tailscale як альтернативу overlay-мережі через Docker Swarm:
- Безкоштовний акаунт на tailscale.com
- Встановлюється на кожну VM як Ansible роль
- Автоматично будує overlay-мережу між VM
- VM можуть комунікувати між собою по іменах без відкриття портів
- Cloudflare Tunnel + Tailscale + авторизація через GitHub — можна зробити так що до вашого сервісу пускає тільки конкретні GitHub акаунти

### Cloudflare Tunnel (бонусна тема)

- Дозволяє виставити сервіс в інтернет без відкриття жодного порту назовні
- DNS challenge для сертифікатів
- Авторизація через GitHub OAuth — хтось відкриває ваш домен, Cloudflare перекидає на GitHub login, якщо email правильний — пускає
